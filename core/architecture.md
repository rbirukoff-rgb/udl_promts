Go Project Architecture Template

Цель

Этот документ определяет обязательные архитектурные правила проекта.

Основные принципы:

- Простота
- Предсказуемость
- Тестируемость
- Отсутствие циклических зависимостей
- Контролируемое управление ресурсами
- Безопасное расширение
- Явные зависимости
- Минимальный публичный API

---

1. Структура проекта

Базовая структура:

/cmd/app
    main.go

/internal/app
    init.go
    shutdown.go

/internal/config
    config.go
    config_test.go

/internal/<module>

/internal/core      (опционально)

/pkg/<utils>        (опционально)

/configs

/docs

/tests

Правила:

1. "cmd/" содержит только точки входа.
2. "internal/" содержит бизнес-логику проекта.
3. "pkg/" используется только для действительно переиспользуемых пакетов.
4. Если пакет нужен только текущему проекту — он располагается внутри "internal/".
5. Каждый пакет решает одну задачу.
6. Циклические зависимости запрещены.
7. Структура проекта должна отражать предметную область.
8. Для значимых пакетов рекомендуется наличие собственных тестов.
9. Конфигурация располагается в отдельном пакете "internal/config".
10. Точка композиции приложения располагается в "cmd" или "internal/app".

---

2. Правила зависимостей

Допустимое направление зависимостей:

cmd
 ↓
app
 ↓
modules
 ↓
utils

Правила:

1. Импорт вверх запрещён.
2. Циклические зависимости запрещены.
3. Зависимости должны быть явными.
4. Скрытые глобальные зависимости запрещены.
5. Каждый слой знает только о нижележащих слоях.
6. Компонент зависит от контракта, а не от конкретной реализации.

---

3. Правило композиции зависимостей

Все зависимости приложения собираются в одном месте.

Допустимые точки композиции:

cmd/main.go

или

internal/app

Правила:

1. Создание зависимостей централизовано.
2. Бизнес-логика не создаёт собственные зависимости.
3. Компоненты получают зависимости через конструкторы.
4. Скрытое создание инфраструктурных объектов запрещено.
5. Композиция должна быть читаемой и очевидной.

Плохо:

func NewUserService() *UserService {
    db := postgres.New()
    cache := redis.New()
    return &UserService{}
}

Хорошо:

db := postgres.New()
cache := redis.New()

svc := user.New(db, cache)

---

4. Правила глобального состояния

Правила:

1. Глобальные изменяемые переменные запрещены.
2. Singleton допускается только для действительно глобальных ресурсов.
3. Зависимости передаются явно через конструкторы.
4. Неявные зависимости запрещены.
5. Предпочтительна явная передача состояния.

---

5. Конфигурация

Правила:

1. Конфигурация загружается один раз при старте.
2. Конфигурация считается неизменяемой.
3. Все настройки собираются в структуру "Config".
4. Использование "os.Getenv()" вне конфигурационного слоя запрещено.
5. Значения по умолчанию задаются явно.
6. Конфигурация должна валидироваться при запуске.
7. Некорректная конфигурация считается ошибкой старта.
8. Доступ к переменным окружения допускается только внутри пакета конфигурации.

---

6. Инициализация

Порядок запуска:

config
↓
logger
↓
dependencies
↓
modules
↓
run

Правила:

1. Инициализация не должна запускать фоновые процессы.
2. Создание объекта и запуск объекта являются разными этапами жизненного цикла.
3. Все зависимости создаются централизованно.
4. Ошибка инициализации должна останавливать запуск.
5. Инициализация должна быть детерминированной и повторяемой.
6. Использование "init()" допускается только для регистрации или простой внутренней инициализации.
7. Запуск бизнес-логики внутри "init()" запрещён.

---

7. Интерфейсы

Правила:

1. Интерфейсы объявляются там, где используются.
2. Интерфейсы должны быть минимальными.
3. Интерфейсы являются контрактами между пакетами.
4. Реализация интерфейса располагается отдельно от его объявления.
5. Не создавать интерфейсы заранее без реальной потребности.

Предпочтительно:

type Storage interface {
    Save(ctx context.Context, data Data) error
}

---

8. Взаимодействие модулей

Правила:
1. Модуль не должен обращаться к внутреннему состоянию другого модуля.
2. Взаимодействие выполняется через контракты.
3. Экспортируется только необходимое.
4. Внутренние детали остаются неэкспортируемыми.
5. Каждый модуль имеет чёткую ответственность.

---

9. Публичный API

Правила:

1. Экспортируется только то, что действительно необходимо.
2. Неэкспортируемые типы предпочтительнее экспортируемых.
3. Публичный API считается контрактом.
4. Изменение публичного API требует оценки совместимости.
5. Чем меньше публичная поверхность пакета, тем проще сопровождение.

---

10. Ошибки

Правила:

1. Ошибки возвращаются вызывающему коду.
2. Ошибки не логируются на каждом уровне.
3. Ошибки оборачиваются контекстом.
4. Паника допустима только при фатальных ошибках.
5. Ошибки должны быть информативными.

Пример:

return fmt.Errorf("save user: %w", err)

---

11. Управление ресурсами

Ресурсом считается любой объект, требующий освобождения.

Примеры:

- файл;
- сокет;
- соединение с БД;
- HTTP-клиент;
- очередь;
- WASM runtime.

Правила:

1. Каждый ресурс имеет владельца.
2. Кто создал ресурс — тот его закрывает.
3. Освобождение ресурсов должно быть гарантировано.
4. Утечки ресурсов считаются критическим дефектом.

---

12. Контексты

Раздел применяется если используется "context".

Правила:

1. Каждая операция ввода-вывода принимает "context".
2. Каждая длительная операция принимает "context".
3. Корневой "context" создаётся в точке входа.
4. Использование "context.Background()" вне точки входа запрещено.
5. Все таймауты должны быть явными.
6. "context" используется для отмены и передачи метаданных запроса.
7. "context" не используется для передачи бизнес-данных.

---

13. Конкурентный доступ

Раздел применяется если данные используются несколькими потоками исполнения.

Правила:

1. Общие данные должны быть защищены.
2. Запись в map без синхронизации запрещена.
3. Atomic использовать только для простых случаев.
4. Предпочитать передачу данных вместо совместного владения.
5. Гонки данных считаются критическим дефектом.

---

14. Каналы

Раздел применяется только если используются каналы.

Правила:

1. Канал создаёт его владелец.
2. Канал закрывает его владелец.
3. Закрытие чужого канала запрещено.
4. Направление владения каналом должно быть очевидным.

---

15. Горутины

Раздел применяется только если используются горутины.

Правила:

1. Каждая горутина должна иметь механизм завершения.
2. Каждая горутина принимает "context".
3. Бесконечные циклы без "select" запрещены.
4. Создание неконтролируемых дочерних горутин запрещено.
5. Количество фоновых процессов должно быть предсказуемым.
6. Для группы связанных горутин рекомендуется использовать "errgroup".

Шаблон:

func worker(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return
        }
    }
}

---

16. Логирование

Правила:

1. Логирование централизовано.
2. Логирование потокобезопасно.
3. Логирование не должно влиять на бизнес-логику.
4. Формат логов единообразен.
5. Уровни логирования определяются проектом.

Если используется асинхронный логгер:

- допускается буферизация;
- допускается fallback-механизм;
- допускается кольцевой буфер.

---

17. Наблюдаемость

Для средних и крупных проектов рекомендуется:

- logs;
- metrics;
- traces.

Правила:

1. Подсистемы наблюдаемости независимы.
2. Отказ метрик не должен ломать приложение.
3. Отказ трассировки не должен ломать приложение.

---

18. Shutdown

Раздел применяется только для долгоживущих приложений.

Примеры:

- сервер;
- демон;
- воркер;
- агент.

Правила:

1. Завершение должно быть контролируемым.
2. Новая работа перестаёт приниматься до остановки.
3. Все фоновые процессы должны завершиться.
4. Все ресурсы должны быть освобождены.
5. Таймаут завершения задаётся явно.

---

19. Тестируемость

Правила:
1. Каждый модуль тестируется изолированно.
2. Внешние зависимости подменяются контрактами.
3. Тесты должны быть воспроизводимыми.
4. Тесты не должны зависеть от порядка запуска.
5. Интеграционные тесты отделяются от модульных.
6. Конфигурация должна иметь собственные тесты.
7. Тесты не должны использовать глобальное состояние.

---

20. Расширения и плагины

Раздел применяется только если проект поддерживает расширения.

Правила:

1. Расширения работают через контракты.
2. Расширения не должны иметь доступ к внутренним пакетам.
3. Ошибка расширения не должна приводить к падению ядра.
4. Расширение должно быть отключаемым.
5. Все точки расширения документируются.

---

21. WASM

Раздел применяется только если используется WebAssembly.

Правила:

1. WASM рассматривается как внешнее расширение.
2. Взаимодействие осуществляется только через Host API.
3. Прямой доступ к внутреннему состоянию запрещён.
4. Выполнение должно иметь таймаут.
5. Использование памяти должно контролироваться.
6. ABI должно иметь версию.
7. Ошибка WASM не должна приводить к падению приложения.

Если WASM не используется — раздел игнорируется.

---

22. Совместимость и версии

Правила:

1. Публичные контракты считаются стабильными.
2. Ломающие изменения требуют новой версии.
3. Форматы обмена данными версионируются.
4. Контракты плагинов версионируются.
5. Контракты WASM версионируются отдельно.

---

Архитектурный чек-лист

Обязательно проверить:

- структура проекта;
- зависимости;
- композиция зависимостей;
- глобальное состояние;
- конфигурация;
- инициализация;
- интерфейсы;
- публичный API;
- взаимодействие модулей;
- ошибки;
- ресурсы;
- тестируемость.

Дополнительно при использовании:

- context;
- goroutines;
- channels;
- logging;
- observability;
- shutdown;
- plugins;
- wasm.
